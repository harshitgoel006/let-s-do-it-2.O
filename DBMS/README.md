# Q1. What is DBMS?

# Interview Answer

**DBMS (Database Management System)** is a software that is used to **create, store, retrieve, update, and manage data** in a database.

It acts as an **interface between the user/application and the database**, making data management easy, secure, and efficient.

**Examples:** MySQL, PostgreSQL, Oracle, SQL Server.

---

# Detailed Explanation

Imagine a college.

Thousands of student records are stored.

Instead of manually searching files, the college uses **MySQL**.

Now they can:

* Add new students
* Update student details
* Delete records
* Search any student in seconds

All these operations are handled by the **DBMS**.

So, a DBMS helps us **organize data and perform operations on it easily**.

---

# Main Functions of DBMS

* Store data
* Retrieve data
* Update data
* Delete data
* Provide Security
* Manage multiple users
* Reduce data redundancy

---

# Real-Life Example

Think about **Instagram**.

When you log in:

* Your username and password are stored in a database.
* DBMS verifies your credentials.
* It fetches your profile, followers, and posts.

Without a DBMS, managing millions of users would be extremely difficult.

---

# Cross Questions

### Q1. What does DBMS stand for?

**Database Management System.**

---

### Q2. Is DBMS a software?

**Yes.** It is software used to manage databases.

---

### Q3. Give some examples of DBMS.

* MySQL
* PostgreSQL
* Oracle
* Microsoft SQL Server

---

### Q4. What are the basic operations of a DBMS?

**Create, Read, Update, Delete (CRUD).**

---

# Q4. What are the Advantages of DBMS?

# Interview Answer

A **DBMS** provides an efficient way to store and manage data. It improves **security, reduces duplicate data, maintains consistency, and allows multiple users to access data simultaneously.**

---

# Detailed Explanation

### 1. Reduces Data Redundancy

The same data is not stored multiple times, which saves storage space.

### 2. Improves Data Security

Only authorized users can access or modify the data.

### 3. Data Consistency

Since duplicate data is reduced, the data remains accurate and consistent.

### 4. Data Sharing

Multiple users can access the same database at the same time.

### 5. Backup & Recovery

DBMS provides backup and recovery features to protect data from loss.

### 6. Easy Data Retrieval

Data can be searched quickly using SQL queries.

---

# Real-Life Example

Think about a **Bank**.

Without a DBMS:

* Customer data may be duplicated.
* Anyone could modify records.
* Recovering lost data would be difficult.

With a DBMS:

* Customer data is secure.
* Duplicate records are minimized.
* Data can be recovered using backups.

# Q5. What are the Disadvantages of DBMS?

# Interview Answer

A **DBMS** has many advantages, but it also has some disadvantages such as **high cost, complexity, large storage requirement, and performance overhead for small applications.**

---

# Detailed Explanation

### 1. High Cost

DBMS software, hardware, and maintenance can be expensive.

### 2. Complex System

A DBMS is more complex than storing data in simple files. It requires trained people to manage it.

### 3. Large Storage Requirement

DBMS itself requires additional memory for logs, indexes, and metadata.

### 4. Performance Overhead

For small applications, a DBMS may be slower than a simple file system because it performs extra operations like security checks and transaction management.

### 5. Failure Can Affect the Whole System

If the DBMS server fails, users may not be able to access the database until it is restored.

---

# Real-Life Example

Suppose a small shop stores only 20 customer records.

Using **MySQL Server** for such a small amount of data may be unnecessary.

A simple Excel sheet may be sufficient and easier to manage.

---

# Cross Questions

### Q1. What is the biggest disadvantage of DBMS?

**High cost and complexity.**

---

### Q2. Is DBMS suitable for every application?

**No.** For very small applications, a file system may be enough.

---

### Q3. Why does a DBMS need more storage?

Because it stores **indexes, logs, metadata, and backup information** in addition to the actual data.

---

# Q6. What is RDBMS?

# Interview Answer

**RDBMS (Relational Database Management System)** is a type of DBMS that stores data in the form of **tables (rows and columns)**. The tables are related to each other using **keys**.

**Examples:** MySQL, PostgreSQL, Oracle, SQL Server.

---

# Detailed Explanation

In RDBMS:

* Data is stored in tables.
* Each row is called a **Record (Tuple)**.
* Each column is called an **Attribute**.
* Tables are connected using **Primary Key** and **Foreign Key**.

Example:

**Student Table**

| ID | Name  | Dept |
| -- | ----- | ---- |
| 1  | Rahul | CSE  |
| 2  | Aman  | IT   |

---

# Real-Life Example

* Users
* Posts
* Comments

These tables are related using IDs.

# Q14. What is TCL (Transaction Control Language)?

# Interview Answer

**TCL (Transaction Control Language)** is used to **manage transactions** in a database.

The main commands are **COMMIT, ROLLBACK, and SAVEPOINT**.

---

# Detailed Explanation

A **transaction** is a group of SQL statements executed as a single unit.

TCL commands:

* `COMMIT` → Permanently save changes.
* `ROLLBACK` → Undo changes.
* `SAVEPOINT` → Create a checkpoint to roll back to.

---

# Example

```sql
UPDATE Student
SET marks = 95
WHERE id = 1;

COMMIT;
```

```sql
UPDATE Student
SET marks = 95
WHERE id = 1;

ROLLBACK;
```

---

# Real-Life Example

Imagine you're transferring **₹5,000** from Account A to Account B.

* If both operations succeed → **COMMIT**
* If any operation fails → **ROLLBACK**

This prevents incorrect account balances.

---

# Cross Questions

### Q1. What is TCL?

### Q2. Which command permanently saves changes?

**COMMIT**

### Q3. Which command undoes changes?

**ROLLBACK**

### Q4. What is SAVEPOINT?

A checkpoint inside a transaction to roll back to if needed.

---

# ⭐ Interview Tip

One of the most common interview questions:

Because until you **COMMIT**, changes are **not permanently saved** (depending on the DBMS and transaction settings).

# Q14. What is TCL (Transaction Control Language)?

## Interview Answer

**TCL (Transaction Control Language)** is a category of SQL commands used to **manage transactions** in a database.

A transaction is a group of SQL statements that execute as **one single unit**. TCL ensures that either **all operations are successfully completed or none of them are applied**, maintaining database consistency.

The main TCL commands are:

* **COMMIT**
* **ROLLBACK**
* **SAVEPOINT**

---

## Detailed Explanation

### What is a Transaction?

A **transaction** is a sequence of one or more SQL operations treated as a **single logical unit of work**.

Example:

```sql
UPDATE Accounts
SET balance = balance - 5000
WHERE id = 101;

UPDATE Accounts
SET balance = balance + 5000
WHERE id = 102;
```

Both queries together form **one transaction**.

If the first query succeeds but the second fails, the database becomes inconsistent.

To avoid this, we use **TCL**.

---

### 1. COMMIT

**COMMIT** permanently saves all the changes made during the current transaction.

After COMMIT, the changes **cannot be undone** using ROLLBACK.

Example:

```sql
UPDATE Employee
SET salary = 60000
WHERE id = 1;

COMMIT;
```

---

### 2. ROLLBACK

**ROLLBACK** cancels all the changes made in the current transaction and restores the database to its previous state.

Example:

```sql
UPDATE Employee
SET salary = 60000
WHERE id = 1;

ROLLBACK;
```

Salary remains unchanged.

---

### 3. SAVEPOINT

A **SAVEPOINT** creates a checkpoint inside a transaction.

Instead of undoing everything, you can roll back only to that checkpoint.

Example:

```sql
SAVEPOINT S1;

UPDATE Employee
SET salary = 70000
WHERE id = 1;

ROLLBACK TO S1;
```

Only the changes made after **S1** are undone.

---

## Real-Life Example

Suppose you're transferring **₹10,000** from Account A to Account B.

Step 1: Deduct ₹10,000 from A.

Step 2: Add ₹10,000 to B.

* If both steps succeed → **COMMIT**
* If step 2 fails → **ROLLBACK**

This ensures that money is never deducted without being credited to the other account.

# Q15. What is a Transaction in DBMS?

## Interview Answer

A **Transaction** is a **sequence of one or more SQL operations that are executed as a single logical unit of work**.

## Detailed Explanation

Transactions solve all these problems.

* Failed
* Aborted
* Terminated

The DBMS is preparing to commit.

---

* System crash
* Constraint violation
* Network issue

The transaction cannot continue.

The transaction has finished completely and all resources are released.

# Q17. What are ACID Properties in DBMS?

## Interview Answer

**ACID** is a set of **four properties** that ensures a transaction is executed **reliably, correctly, and consistently**.

The four ACID properties are:

* **A → Atomicity**
* **C → Consistency**
* **I → Isolation**
* **D → Durability**

These properties ensure that even if a system crashes or multiple users access the database simultaneously, the data remains correct and reliable.

---

## Detailed Explanation

### 1. Atomicity (All or Nothing)

Atomicity means a transaction is treated as **one indivisible unit**.

Either **all operations are executed successfully** or **none of them are executed**.

There is no partial execution.

### Example

Suppose you transfer **₹5000** from Account A to Account B.

Transaction:

```sql
UPDATE Accounts
SET balance = balance - 5000
WHERE id = 101;

UPDATE Accounts
SET balance = balance + 5000
WHERE id = 102;
```

If money is deducted from A but cannot be credited to B due to a server crash,

➡️ **ROLLBACK** happens.

Database returns to the previous state.

This is **Atomicity**.

---

### 2. Consistency

Consistency means a transaction always takes the database from **one valid state to another valid state**.

It never leaves the database in an inconsistent state.

### Example

Suppose total money in the bank is:

```
A = ₹10000
B = ₹5000

Total = ₹15000
```

After transferring ₹2000,

```
A = ₹8000
B = ₹7000

Total = ₹15000
```

Total money remains the same.

Database remains consistent.

---

### 3. Isolation

Isolation means when multiple transactions are running simultaneously, **they should not interfere with each other**.

Each transaction behaves as if it is executing alone.

### Example

Suppose:

Rahul is transferring money.

At the same time,

Aman is checking the balance.

Aman should not see the **half-completed transaction**.

He should either see:

* Old balance
* New balance

Never an intermediate balance.

---

### 4. Durability

Durability means once a transaction is **COMMITTED**, the changes are **permanently saved**.

Even if the system crashes immediately after COMMIT,

the data should still exist.

### Example

You place an order on Amazon.

Payment succeeds.

Order is confirmed.

Suddenly electricity goes off.

When the server restarts,

your order is **still present**.

That's Durability.

---

## Summary Table

| Property    | Meaning                      |
| ----------- | ---------------------------- |
| Atomicity   | All or Nothing               |
| Consistency | Valid State → Valid State    |
| Isolation   | Transactions don't interfere |
| Durability  | Committed data is permanent  |

---

## Real-Life Example

Think about an **ATM**.

You withdraw ₹10,000.

The ATM performs:

* Check balance
* Deduct amount
* Dispense cash
* Update balance

If cash is not dispensed due to an error,

the balance should also **not be deducted**.

This follows **Atomicity**.

The account balance remains correct (**Consistency**).

Other users cannot see the transaction in between (**Isolation**).

Once completed, the balance remains updated forever (**Durability**).

# Q19. What is Concurrency Control in DBMS?

**Concurrency Control** is a mechanism in DBMS that ensures **multiple transactions can execute simultaneously without causing data inconsistency**.

# Q20. What is Lost Update Problem?

## Interview Answer

The **Lost Update Problem** occurs when **two transactions update the same data simultaneously**, and one transaction's update **overwrites** the other's update.

As a result, one update is **lost**.

---

## Detailed Explanation

Suppose the current balance is **₹10,000**.

Two transactions start at the same time.

### Transaction T1

Reads ₹10,000 and withdraws ₹2,000.

New Balance = ₹8,000

---

Also reads ₹10,000 and deposits ₹5,000.

New Balance = ₹15,000

---

Now suppose:

* T1 writes ₹8,000.
* Then T2 writes ₹15,000.

Final balance becomes **₹15,000**.

But the correct balance should have been:

```text
10000 - 2000 + 5000 = ₹13,000
```

The withdrawal of ₹2,000 is completely lost.

This is called the **Lost Update Problem**.

DBMS uses:

* Locks
* Timestamp Ordering
* Concurrency Control Protocols

to ensure one transaction completes before another modifies the same data.

---

## Real-Life Example

Suppose two employees update the same salary record simultaneously.

* HR changes salary to ₹60,000.
* Manager changes salary to ₹65,000.

If one update overwrites the other, one employee's work is lost.

# Q21. What is Dirty Read?

## Interview Answer

A **Dirty Read** occurs when one transaction **reads data that has been modified by another transaction but not yet committed**.

If the second transaction is rolled back, the first transaction has read **invalid (dirty) data**.

---

## Detailed Explanation

Suppose there are two transactions:

* **T1** updates an employee's salary from **₹50,000 → ₹60,000**.
* But **T1 has not executed COMMIT yet.**

At the same time,

* **T2** reads the salary and sees **₹60,000**.

Now imagine T1 fails and executes **ROLLBACK**.

The salary becomes **₹50,000** again.

But T2 already used the incorrect value **₹60,000**.

This is called a **Dirty Read**.

---

### Example

Current Salary = **₹50,000**

```text
T1:
UPDATE Salary = 60000
(Not Committed Yet)

↓

T2:
Reads Salary = 60000

↓

T1:
ROLLBACK

Final Salary = 50000
```

T2 read data that never actually existed permanently.

---

## Real-Life Example

Suppose your bank app temporarily shows **₹1,00,000** because another transaction updated your account.

Before the transaction is committed, it fails.

Your actual balance becomes **₹50,000** again.

You saw incorrect information.

That's a **Dirty Read**.

---

Reading **uncommitted data**.

### Q2. Why is it called Dirty?

Because the data is **not permanently saved**.

### Q3. Which command makes Dirty Read impossible?

**COMMIT** (or appropriate isolation levels).

# Q26. Difference Between Schema and Instance ⭐⭐⭐

## Interview Answer

Schema is **static**, while an instance is **dynamic**.

---

```text

---------
id
name
age
```

```text

2  Aman    22
```

Tomorrow another student joins:

```text
1  Harshit 21
2  Aman    22
3  Rahul   20
```

The **Schema remains the same**, but the **Instance changes** because the data has changed.

---

# Q27. What is Data Abstraction in DBMS?

## Interview Answer

**Data Abstraction** is the process of **hiding unnecessary details and showing only the required information to the user**.

Its main objective is to **reduce complexity, improve security, and make the database easier to use**.

In DBMS, Data Abstraction is achieved using **three levels: Physical Level, Logical Level, and View Level.**

---

## Detailed Explanation

In a database, there is a lot of information about **how data is stored, organized, and retrieved**.

Every user doesn't need to know all these details.

For example:

* A customer using a banking app only wants to see **Account Balance**.
* The customer doesn't need to know:

  * Where the data is stored.
  * Which index is used.
  * Which disk block contains the data.

These internal details are hidden. This concept is called **Data Abstraction**.

---

## Why Do We Need Data Abstraction?

Without Data Abstraction:

* Database becomes difficult to understand.
* Security decreases because everyone can see everything.
* Users have to know internal implementation.

With Data Abstraction:

* Users see only what they need.
* Internal implementation remains hidden.
* Database becomes secure and easy to maintain.

---

## Real-Life Example

Think about driving a **car**.

As a driver, you only use:

* Steering
* Brake
* Accelerator

You don't need to know:

* How the engine works.
* How fuel is injected.
* How the gearbox is designed.

The car hides all these complexities.

Similarly, a DBMS hides its internal implementation from users.

---

## Cross Questions

### Q1. What is Data Abstraction?

It is the process of hiding unnecessary details and showing only required information.

### Q2. Why is Data Abstraction needed?

To reduce complexity, improve security, and simplify database usage.

### Q3. How many levels of Data Abstraction are there?

Three.

---

## ⭐ Interview Tip

Many students confuse **Data Abstraction** with **Data Hiding**.

Remember:

* **Data Hiding** is an OOP concept.
* **Data Abstraction** in DBMS means hiding **storage and implementation details**, not data itself.

---

## 📌 Remember Keywords

* Hide Complexity
* Required Information
* Security
* Simplicity
* Three Levels

---

## 📝 Revision (1-Minute)

* Data Abstraction hides unnecessary details.
* Users see only required information.
* Improves security.
* Reduces complexity.
* Has three levels.

# Q28. Explain the Three Levels of Data Abstraction

## Interview Answer

DBMS provides **three levels of Data Abstraction**:

1. **Physical Level (Internal Level)**
2. **Logical Level (Conceptual Level)**
3. **View Level (External Level)**

These levels hide database complexity from different types of users.

---

## Detailed Explanation

### 1. Physical Level (Internal Level)

This is the **lowest level** of abstraction.

It describes **how data is actually stored** inside the database.

It includes:

* Storage location
* File organization
* Indexes
* Disk blocks
* Memory management

This level is mainly handled by the **DBMS and Database Administrator (DBA)**.

**Example:**
How employee records are stored on disk using B+ Trees or indexes.

---

### 2. Logical Level (Conceptual Level)

This is the **middle level**.

It describes **what data is stored** and **how different tables are related**.

It defines:

* Tables
* Columns
* Data Types
* Relationships
* Constraints

This level is used by **database designers and developers**.

**Example:**

```text
Student
---------
ID
Name
Age

Department
-----------
DeptID
DeptName
```

It defines the structure but not where the data is physically stored.

---

### 3. View Level (External Level)

This is the **highest level**.

It describes **what part of the database a particular user can see**.

Different users can have different views of the same database.

**Example:**

In a college:

* Student → Can view marks.
* Faculty → Can update marks.
* Principal → Can view all student records.

Each user sees only the required information.

---

## Diagram

```text
            Users
               │
               ▼
      View Level (External)
               │
               ▼
   Logical Level (Conceptual)
               │
               ▼
   Physical Level (Internal)
```

---

## Real-Life Example

Think of **Instagram**.

* **View Level:** You see posts, likes, comments.
* **Logical Level:** Developers know there are Users, Posts, Followers tables.
* **Physical Level:** The server knows exactly where and how data is stored on disks.

---

## Cross Questions

### Q1. Which is the highest level?

**View Level**

### Q2. Which level stores data physically?

**Physical Level**

### Q3. Which level defines tables and relationships?

**Logical Level**

### Q4. Which users use the View Level?

End users.

### Q5. Which level is handled by the DBA?

Physical Level.

---

## ⭐ Interview Tip

A very common interview question is:

> **Difference between Physical and Logical Level?**

* **Physical Level** → *How* data is stored.
* **Logical Level** → *What* data is stored.

Just remember:

* **How = Physical**
* **What = Logical**
* **Who can see = View**

---

## 📌 Remember Keywords

* Physical
* Logical
* View
* Internal
* Conceptual
* External

---

## 📝 Revision (1-Minute)

* DBMS has three abstraction levels.
* Physical → How data is stored.
* Logical → What data is stored.
* View → What users can see.
* Improves security and reduces complexity.

---

## 🎯 Memory Trick

Remember the sequence:

**P → L → V**

* **P** = Physical = **How**
* **L** = Logical = **What**
* **V** = View = **Who Sees What**

# Q29. What is a Super Key?

## Interview Answer

A **Super Key** is a **set of one or more attributes that can uniquely identify each record in a table**.

It may contain **extra (unnecessary) attributes**, but it must uniquely identify every row.

---

## Detailed Explanation

Consider the following **Student** table:

| Roll No | Aadhaar No | Email                             | Name  |
| ------- | ---------- | --------------------------------- | ----- |
| 101     | 1111       | [a@gmail.com](mailto:a@gmail.com) | Rahul |
| 102     | 2222       | [b@gmail.com](mailto:b@gmail.com) | Aman  |

Possible Super Keys are:

* Roll No ✅
* Aadhaar No ✅
* Email ✅
* Roll No + Name ✅
* Aadhaar No + Email ✅
* Roll No + Email + Name ✅

All of these uniquely identify a student, so they are **Super Keys**.

Notice that **Roll No + Name** is still a Super Key even though **Name is unnecessary**.

---

## Characteristics

* Uniquely identifies every row.
* Can contain extra attributes.
* Every Candidate Key is a Super Key.
* Not every Super Key is a Candidate Key.

---

## Real-Life Example

In a college:

A student can be uniquely identified by:

* Roll Number
* Aadhaar Number
* College Email

Even **Roll Number + Name** identifies the student, but **Name is extra**.

---

## Cross Questions

### Q1. Can a Super Key contain extra attributes?

**Yes.**

### Q2. Can there be multiple Super Keys?

**Yes.**

### Q3. Is every Super Key a Candidate Key?

**No.**

---

## ⭐ Interview Tip

Remember:

> **Super Key = Unique + Extra Attributes Allowed**

---

## 📌 Remember Keywords

* Unique
* Extra Attributes
* Multiple
* Identification
* Superset

---

## 📝 Revision

* Super Key uniquely identifies records.
* Extra attributes are allowed.
* Multiple Super Keys can exist.
* Candidate Keys are derived from Super Keys.

# Q30. What is a Candidate Key?

## Interview Answer

A **Candidate Key** is the **smallest possible Super Key** that uniquely identifies each record.

It does **not contain any unnecessary attributes**.

---

## Detailed Explanation

Using the same Student table:

| Roll No | Aadhaar No | Email                             | Name  |
| ------- | ---------- | --------------------------------- | ----- |
| 101     | 1111       | [a@gmail.com](mailto:a@gmail.com) | Rahul |
| 102     | 2222       | [b@gmail.com](mailto:b@gmail.com) | Aman  |

Super Keys:

* Roll No
* Aadhaar No
* Email
* Roll No + Name
* Roll No + Email

Candidate Keys:

* Roll No
* Aadhaar No
* Email

Because these are already sufficient.

**Roll No + Name** is **not** a Candidate Key because Name is unnecessary.

---

## Characteristics

* Minimal Super Key.
* No extra attributes.
* Multiple Candidate Keys can exist.
* One Candidate Key is chosen as the Primary Key.

---

## Real-Life Example

For an employee:

* Employee ID
* PAN Number
* Aadhaar Number

All three uniquely identify an employee.

Each one is a **Candidate Key**.

---

## Cross Questions

### Q1. What is the difference between Super Key and Candidate Key?

Super Key may have extra attributes.

Candidate Key never has extra attributes.

### Q2. Can a table have multiple Candidate Keys?

**Yes.**

### Q3. How is the Primary Key selected?

One Candidate Key is selected as the Primary Key.

---

## ⭐ Interview Tip

This is the easiest way to remember:

> **Candidate Key = Minimal Super Key**

---

## 📌 Remember Keywords

* Minimal
* Unique
* No Extra Attribute
* Multiple Candidate Keys

---

## 📝 Revision

* Candidate Key uniquely identifies rows.
* No unnecessary attributes.
* Smallest Super Key.
* One Candidate Key becomes Primary Key.

# Q31. What is a Primary Key?

## Interview Answer

A **Primary Key** is the **Candidate Key selected to uniquely identify each record in a table**.

A Primary Key **cannot contain NULL values** and **must always be unique**.

Each table can have **only one Primary Key**.

---

## Detailed Explanation

Example:

| Roll No | Name  | Age |
| ------- | ----- | --- |
| 101     | Rahul | 20  |
| 102     | Aman  | 21  |

Here,

**Roll No** is chosen as the Primary Key because:

* Every student has a unique Roll Number.
* It never changes.
* It cannot be NULL.

---

## Characteristics

* Uniquely identifies each row.
* Cannot be NULL.
* Cannot contain duplicate values.
* Only one Primary Key per table.

---

### SQL Example

```sql
CREATE TABLE Student(
    RollNo INT PRIMARY KEY,
    Name VARCHAR(50),
    Age INT
);
```

---

## Real-Life Example

Think of your **Aadhaar Number**.

Every citizen has a unique Aadhaar number.

No two people can share the same Aadhaar number.

Similarly, every record has one Primary Key.

---

## Cross Questions

### Q1. Can a Primary Key contain NULL?

**No.**

### Q2. Can two rows have the same Primary Key?

**No.**

### Q3. How many Primary Keys can a table have?

**Only one.**

### Q4. Can a Primary Key be changed?

Technically yes, but it is generally avoided because many tables may reference it.

---

## ⭐ Interview Tip

One of the most common interview questions:

> **Can a Candidate Key contain NULL?**

Theoretically, a candidate key should uniquely identify a row and is chosen to become a primary key, so in practice **candidate keys are also expected to be unique and not NULL**.

---

## 📌 Remember Keywords

* Unique
* Not NULL
* One Per Table
* Candidate Key
* Identifier

---

## 📝 Revision

* Primary Key is a selected Candidate Key.
* Must be unique.
* Cannot contain NULL.
* Only one Primary Key per table.
* Used to identify every record.

# Q33. What is a Foreign Key?

## Interview Answer

A **Foreign Key** is a column (or group of columns) in one table that **refers to the Primary Key of another table**.

It is used to **create relationships between tables** and maintain **Referential Integrity**.

---

## Detailed Explanation

Consider two tables:

### Department

| DeptID | DeptName |
| ------ | -------- |
| 1      | CSE      |
| 2      | IT       |

Here,

**DeptID** is the **Primary Key**.

---

### Student

| RollNo | Name  | DeptID |
| ------ | ----- | ------ |
| 101    | Rahul | 1      |
| 102    | Aman  | 2      |

Here,

**DeptID** is a **Foreign Key** because it refers to the **Department** table.

Relationship:

```text
Department
------------
DeptID (PK)
DeptName

        ▲
        │
Student
------------
RollNo (PK)
Name
DeptID (FK)
```

---

## Why Do We Need Foreign Keys?

Foreign Keys ensure that:

* Only valid data is inserted.
* Relationships between tables are maintained.
* Orphan records are avoided.

For example, you cannot assign **DeptID = 10** if no such department exists.

---

## SQL Example

```sql
CREATE TABLE Department(
    DeptID INT PRIMARY KEY,
    DeptName VARCHAR(50)
);

CREATE TABLE Student(
    RollNo INT PRIMARY KEY,
    Name VARCHAR(50),
    DeptID INT,
    FOREIGN KEY (DeptID)
    REFERENCES Department(DeptID)
);
```

---

## Real-Life Example

Think of **Amazon Orders**.

* Customer table has **CustomerID (PK)**.
* Orders table stores **CustomerID (FK)**.

This tells us **which customer placed which order**.

---

## Cross Questions

### Q1. What does a Foreign Key reference?

The Primary Key of another table.

### Q2. Can a table have multiple Foreign Keys?

**Yes.**

### Q3. Why do we use Foreign Keys?

To create relationships and maintain referential integrity.

### Q4. Can a Foreign Key contain duplicate values?

**Yes.** Many students can belong to the same department.

---

## ⭐ Interview Tip

A very common question:

> **Can a Foreign Key contain NULL?**

✅ **Yes**, unless the column is explicitly defined as **NOT NULL**.

Example:

A student may not yet be assigned to any department, so the Foreign Key can be NULL.

---

## 📌 Remember Keywords

* Relationship
* References
* Primary Key
* Referential Integrity
* Parent Table
* Child Table

---

## 📝 Revision

* Foreign Key references another table.
* Creates relationships.
* Maintains referential integrity.
* Duplicate values are allowed.
* NULL values are allowed unless restricted.

# Q34. What is a Composite Key?

## Interview Answer

A **Composite Key** is a key formed by **combining two or more columns** to uniquely identify a record.

It is used when a **single column is not sufficient** to identify rows uniquely.

---

## Detailed Explanation

Consider the following table:

| StudentID | CourseID | Marks |
| --------- | -------- | ----- |
| 101       | C101     | 90    |
| 101       | C102     | 85    |
| 102       | C101     | 88    |

Here,

* StudentID alone is **not unique**.
* CourseID alone is **not unique**.

But together:

**(StudentID + CourseID)** uniquely identify each record.

So they form a **Composite Key**.

---

## SQL Example

```sql
CREATE TABLE StudentCourse(
    StudentID INT,
    CourseID INT,
    Marks INT,
    PRIMARY KEY(StudentID, CourseID)
);
```

---

## Real-Life Example

In an online learning platform:

A student can enroll in many courses.

A course can have many students.

The combination of **StudentID + CourseID** uniquely identifies one enrollment.

---

## Cross Questions

### Q1. What is a Composite Key?

A key made up of two or more columns.

### Q2. Why is it used?

When a single column cannot uniquely identify records.

### Q3. Can a Composite Key be a Primary Key?

**Yes.**

---

## ⭐ Interview Tip

Don't confuse:

* **Composite Key** → One key made of multiple columns.
* **Multiple Primary Keys** → ❌ Not possible.

A table can have **only one Primary Key**, but that Primary Key can be **composite**.

---

## 📌 Remember Keywords

* Combination
* Multiple Columns
* Unique
* Composite Primary Key

---

## 📝 Revision

* Composite Key = Multiple columns.
* Used when one column is insufficient.
* Can act as a Primary Key.

# Q35. What is a Unique Key?

## Interview Answer

A **Unique Key** is a constraint used to ensure that **all values in a column are unique**.

Unlike a Primary Key, a **Unique Key can contain NULL values** (the exact behavior depends on the DBMS, but in interviews you can say it generally allows NULL).

A table can have **multiple Unique Keys**.

---

## Detailed Explanation

Suppose we have an Employee table:

| EmpID | Email                             | Phone      |
| ----- | --------------------------------- | ---------- |
| 101   | [a@gmail.com](mailto:a@gmail.com) | 9876543210 |
| 102   | [b@gmail.com](mailto:b@gmail.com) | 9876543211 |

If **Email** is declared as a **UNIQUE** column:

* Two employees cannot have the same email.
* Every email must be different.
* One (or sometimes more, depending on the DBMS) NULL values may be allowed.

---

### SQL Example

```sql id="wq18av"
CREATE TABLE Employee(
    EmpID INT PRIMARY KEY,
    Email VARCHAR(100) UNIQUE
);
```

---

## Characteristics

* Prevents duplicate values.
* NULL is generally allowed.
* Multiple Unique Keys can exist.
* Mainly used for alternate unique attributes like Email, PAN, Passport No.

---

## Real-Life Example

In Gmail,

Two users **cannot have the same email address**.

Every email must be unique.

Hence, Email behaves like a **Unique Key**.

# Q42. What is Cardinality in DBMS?

## Interview Answer

**Cardinality** defines **how many instances of one entity can be associated with another entity**.

In simple words, it tells us **the number of relationships between two entities**.

The four main types are:

* One-to-One (1:1)
* One-to-Many (1:N)
* Many-to-One (N:1)
* Many-to-Many (M:N)

---

## Detailed Explanation

Cardinality answers questions like:

* Can one student enroll in many courses?
* Can one department have many employees?
* Can one customer place many orders?

Depending on the answer, we decide the relationship type.

---

### 1. One-to-One (1:1)

One entity is related to only one entity.

Example:

```text
Person -------- Passport
```

* One Person → One Passport
* One Passport → One Person

---

### 2. One-to-Many (1:N)

One record can be associated with multiple records.

Example:

```text
Department -------- Employee
```

One Department

↓

Many Employees

Example:

```text
CSE
 ├── Rahul
 ├── Aman
 └── Priya
```

This is the **most common relationship** in databases.

---

### 3. Many-to-One (N:1)

Many records are associated with one record.

Example:

Many Employees

↓

One Department

```text
Rahul ─┐
Aman ──┼──► CSE
Priya ─┘
```

This is simply the reverse view of One-to-Many.

---

### 4. Many-to-Many (M:N)

Many records from one entity are related to many records of another entity.

Example:

```text
Student -------- Course
```

* One Student studies many Courses.
* One Course has many Students.

```text
Rahul → DBMS
Rahul → Java

Aman → DBMS
Aman → OS
```

This is a **Many-to-Many** relationship.

In relational databases, this is implemented using a **junction (bridge) table**.

---

## Real-Life Example

Amazon:

* One Customer → Many Orders (**1:N**)
* One Order → One Customer (**N:1**)
* One Order → Many Products
* One Product → Many Orders

So **Order ↔ Product** is **Many-to-Many**.

---

## Cross Questions

### Q1. What is Cardinality?

It defines the number of relationships between entities.

### Q2. Which cardinality is most common?

**One-to-Many (1:N).**

### Q3. Which relationship requires a junction table?

**Many-to-Many (M:N).**

---

## ⭐ Interview Tip

Very common interviewer question:

> **Can MySQL directly implement a Many-to-Many relationship?**

**No.**

It is implemented using a **third table (junction/bridge table)** containing the Foreign Keys of both tables.

Example:

```text
StudentCourse
--------------
StudentID (FK)
CourseID (FK)
```

---

## 📌 Remember Keywords

* Cardinality
* One-to-One
* One-to-Many
* Many-to-One
* Many-to-Many
* Junction Table

---

## 📝 Revision (1-Minute)

* Cardinality tells how many entities are related.
* Four types of relationships.
* 1:N is the most common.
* M:N requires a junction table.

# Q45. What is Normalization?

## Interview Answer

**Normalization** is the process of **organizing data in a database to reduce redundancy (duplicate data) and eliminate anomalies**, thereby improving data integrity and consistency.

It divides large tables into smaller related tables and connects them using keys.

---

## Detailed Explanation

Suppose we have the following table:

| StudentID | StudentName | Course | Faculty |
| --------- | ----------- | ------ | ------- |
| 101       | Rahul       | DBMS   | Amit    |
| 101       | Rahul       | Java   | Rohit   |
| 102       | Aman        | DBMS   | Amit    |

Notice that:

* Rahul's name is repeated.
* Faculty "Amit" is repeated.

This repeated information is called **Data Redundancy**.

Normalization removes such unnecessary repetition.

After normalization:

### Student Table

| StudentID | StudentName |
| --------- | ----------- |
| 101       | Rahul       |
| 102       | Aman        |

### Course Table

| CourseID | Course | Faculty |
| -------- | ------ | ------- |
| 1        | DBMS   | Amit    |
| 2        | Java   | Rohit   |

### Enrollment Table

| StudentID | CourseID |
| --------- | -------- |
| 101       | 1        |
| 101       | 2        |
| 102       | 1        |

Now the data is cleaner and duplication is reduced.

---

## Objectives of Normalization

* Reduce data redundancy.
* Remove anomalies.
* Improve consistency.
* Improve data integrity.
* Make maintenance easier.

---

## Real-Life Example

Think of a school register.

Instead of writing the school name in every student's record, you store it once and let all student records refer to it.

This avoids unnecessary repetition.

---

# Q47. What are Anomalies in DBMS?

## Interview Answer

**Anomalies** are **problems that occur due to poor database design and data redundancy**.

There are three main types:

1. Insertion Anomaly
2. Update Anomaly
3. Deletion Anomaly

---

## Detailed Explanation

Consider this table:

| Student | Course | Faculty |
| ------- | ------ | ------- |
| Rahul   | DBMS   | Amit    |
| Rahul   | Java   | Rohit   |
| Aman    | DBMS   | Amit    |

Let's see the problems.

---

### 1. Insertion Anomaly

Suppose a new course **Python** starts, but no student has enrolled yet.

Where will you store it?

You **can't**, because the table requires student details too.

This is an **Insertion Anomaly**.

---

### 2. Update Anomaly

Suppose Faculty **Amit** changes his name to **Dr. Amit**.

You must update every row where "Amit" appears.

If you miss one row, inconsistent data is created.

This is an **Update Anomaly**.

---

### 3. Deletion Anomaly

Suppose Rahul leaves the Java course.

If Rahul was the **only student** in Java, deleting his record also removes the information that **Rohit teaches Java**.

Important information is lost.

This is a **Deletion Anomaly**.

---

## Real-Life Example

Imagine an employee database where the department name is repeated in every employee record.

If the department name changes, you must update hundreds of rows.

That's an **Update Anomaly**.

---

## Cross Questions

### Q1. How many anomalies are there?

Three.

### Q2. Which anomaly causes inconsistent data?

**Update Anomaly.**

### Q3. Which anomaly causes loss of data?

**Deletion Anomaly.**

### Q4. Which anomaly prevents inserting incomplete information?

**Insertion Anomaly.**

## ⭐ Interview Tip

# Q48. What is First Normal Form (1NF)?

## Interview Answer

A table is in **First Normal Form (1NF)** if:

* Every column contains **atomic (single) values**.
* There are **no repeating groups or multi-valued attributes**.
* Each row is unique.

---

## Detailed Explanation

Suppose we have this table:

| StudentID | Name  | Phone Numbers |
| --------- | ----- | ------------- |
| 101       | Rahul | 98765, 87654  |
| 102       | Aman  | 91234         |

This table is **NOT in 1NF** because one cell contains **multiple phone numbers**.

### Convert to 1NF

| StudentID | Name  | Phone Number |
| --------- | ----- | ------------ |
| 101       | Rahul | 98765        |
| 101       | Rahul | 87654        |
| 102       | Aman  | 91234        |

Now each cell has only one value.

Hence, the table is in **1NF**.

---

## Rules of 1NF

* Atomic values only.
* No multi-valued attributes.
* No repeating groups.
* Every row should be uniquely identifiable.

---

## Real-Life Example

A student's record should store **one phone number per row** instead of multiple numbers in one column.

---

## Cross Questions

### Q1. What is the main rule of 1NF?

Every attribute must contain **only one value**.

### Q2. Can a column contain multiple phone numbers in 1NF?

**No.**

### Q3. What are atomic values?

Values that **cannot be divided further**.

---

## ⭐ Interview Tip

Many students think 1NF removes duplication.

❌ Wrong.

1NF only ensures **atomic values**.

Duplicate data may still exist.

---

## 📌 Remember Keywords

* Atomic
* Single Value
* No Repeating Group
* No Multi-valued Attribute

---

## 📝 Revision

* One value per cell.
* No comma-separated values.
* No multi-valued attributes.
* Duplicate data may still exist.

# Q49. What is Second Normal Form (2NF)?

## Interview Answer

A table is in **Second Normal Form (2NF)** if:

1. It is already in **1NF**.
2. Every non-key attribute is **fully dependent on the entire Primary Key**.
3. There should be **no Partial Dependency**.

---

## What is Partial Dependency?

A **Partial Dependency** occurs when a non-key attribute depends on **only part of a composite primary key**, instead of the whole key.

This happens **only when the Primary Key is composite**.

---

## Detailed Example

Suppose we have:

| StudentID | CourseID | StudentName | CourseName |
| --------- | -------- | ----------- | ---------- |
| 101       | DBMS     | Rahul       | Database   |
| 101       | Java     | Rahul       | Java       |
| 102       | DBMS     | Aman        | Database   |

Primary Key = **(StudentID, CourseID)**

Problem:

* StudentName depends only on **StudentID**.
* CourseName depends only on **CourseID**.

Neither depends on the entire composite key.

This is **Partial Dependency**.

---

### Convert to 2NF

### Student

| StudentID | StudentName |
| --------- | ----------- |
| 101       | Rahul       |
| 102       | Aman        |

### Course

| CourseID | CourseName |
| -------- | ---------- |
| DBMS     | Database   |
| Java     | Java       |

### Enrollment

| StudentID | CourseID |
| --------- | -------- |
| 101       | DBMS     |
| 101       | Java     |
| 102       | DBMS     |

Now every non-key attribute depends on its table's key.

---

## Real-Life Example

A student's name depends only on the student, not on which course they take.

So it should be stored separately.

---

## Cross Questions

### Q1. What does 2NF remove?

**Partial Dependency.**

### Q2. Is 1NF required before 2NF?

**Yes.**

### Q3. Can Partial Dependency exist with a single-column Primary Key?

**No.**

---

## ⭐ Interview Tip

A favorite interview question:

> **Can a table with a single Primary Key violate 2NF?**

**No.**

Because Partial Dependency is only possible with a **Composite Primary Key**.

---

## 📌 Remember Keywords

* Partial Dependency
* Composite Key
* Fully Dependent
* 1NF Required

---

## 📝 Revision

* 2NF = 1NF + No Partial Dependency.
* Partial Dependency occurs only with Composite Keys.
* Split the table to remove Partial Dependency.

---

# Q50. What is Third Normal Form (3NF)?

## Interview Answer

A table is in **Third Normal Form (3NF)** if:

1. It is already in **2NF**.
2. There is **no Transitive Dependency**.

---

## What is Transitive Dependency?

A **Transitive Dependency** occurs when a **non-key attribute depends on another non-key attribute**, instead of depending directly on the Primary Key.

---

## Detailed Example

| EmpID | EmpName | DeptID | DeptName |
| ----- | ------- | ------ | -------- |
| 1     | Rahul   | 10     | Sales    |
| 2     | Aman    | 20     | HR       |

Primary Key = **EmpID**

Dependencies:

* EmpName depends on EmpID ✅
* DeptID depends on EmpID ✅
* DeptName depends on DeptID ❌

Here,

DeptName depends on another non-key attribute (DeptID), not directly on EmpID.

This is **Transitive Dependency**.

---

### Convert to 3NF

### Employee

| EmpID | EmpName | DeptID |
| ----- | ------- | ------ |
| 1     | Rahul   | 10     |
| 2     | Aman    | 20     |

### Department

| DeptID | DeptName |
| ------ | -------- |
| 10     | Sales    |
| 20     | HR       |

Now DeptName depends on **DeptID**, which is the Primary Key of the Department table.

No transitive dependency remains.

---

## Real-Life Example

Instead of storing the department name with every employee, create a separate Department table and link it using **DeptID**.

---

## Cross Questions

### Q1. What does 3NF remove?

**Transitive Dependency.**

### Q2. Is 2NF required before 3NF?

**Yes.**

### Q3. What is the difference between Partial and Transitive Dependency?

* **Partial Dependency:** Non-key attribute depends on part of a composite key.
* **Transitive Dependency:** Non-key attribute depends on another non-key attribute.

---

## ⭐ Interview Tip

A very common interview question:

> **Difference between 2NF and 3NF?**

* **2NF removes Partial Dependency.**
* **3NF removes Transitive Dependency.**

---

## 📌 Remember Keywords

* Transitive Dependency
* Non-Key Attribute
* Depends on Another Non-Key
* 2NF Required

---

## 📝 Revision

* 3NF = 2NF + No Transitive Dependency.
* Non-key attributes should depend only on the Primary Key.
* Remove indirectly dependent attributes into separate tables.

# Q51. What is BCNF (Boyce-Codd Normal Form)?

## Interview Answer

**BCNF (Boyce-Codd Normal Form)** is an **improved version of 3NF**.

A table is in BCNF if:

> **For every Functional Dependency (X → Y), X must be a Super Key.**

In simple words:

**Only a Super Key should determine other attributes.**

---

## Detailed Explanation

Consider the table:

| Student | Subject | Teacher |
| ------- | ------- | ------- |
| Rahul   | DBMS    | Amit    |
| Aman    | DBMS    | Amit    |
| Rahul   | Java    | Rohit   |

Assume:

* A teacher teaches only one subject.
* One subject can have multiple students.

Functional Dependencies:

```
(Student, Subject) → Teacher
Teacher → Subject
```

Problem:

Teacher determines Subject.

But **Teacher is NOT a Super Key**.

Hence,

The table is in **3NF** but **NOT in BCNF**.

---

### Convert into BCNF

### Teacher Table

| Teacher | Subject |
| ------- | ------- |
| Amit    | DBMS    |
| Rohit   | Java    |

### Student Enrollment

| Student | Teacher |
| ------- | ------- |
| Rahul   | Amit    |
| Aman    | Amit    |
| Rahul   | Rohit   |

Now every determinant is a Super Key.

---

## Real-Life Example

A professor always teaches one subject.

Instead of repeating the subject for every student, store it once with the professor.

---

## Cross Questions

### Q1. Is BCNF stricter than 3NF?

**Yes.**

### Q2. What is the BCNF rule?

Every determinant must be a Super Key.

### Q3. Does every BCNF table satisfy 3NF?

**Yes.**

---

## ⭐ Interview Tip

A favorite question:

> **Difference between 3NF and BCNF?**

* **3NF** allows some dependencies where the determinant is not a Super Key (under certain conditions).
* **BCNF** does not.

So,

**BCNF is stricter than 3NF.**

---

## 📌 Remember Keywords

* Super Key
* Functional Dependency
* Determinant
* Stricter than 3NF

---

## 📝 Revision

* BCNF = Improved 3NF.
* Every determinant must be a Super Key.
* Removes anomalies left in 3NF.

---

# Q52. What is Fourth Normal Form (4NF)?

## Interview Answer

A table is in **Fourth Normal Form (4NF)** if:

* It is already in **BCNF**.
* It has **no Multi-Valued Dependency (MVD)**.

---

## What is Multi-Valued Dependency?

A Multi-Valued Dependency occurs when **one attribute independently determines multiple values of another attribute**.

---

## Example

| Student | Hobby    | Language |
| ------- | -------- | -------- |
| Rahul   | Cricket  | Hindi    |
| Rahul   | Cricket  | English  |
| Rahul   | Football | Hindi    |
| Rahul   | Football | English  |

Problems:

Student has:

* Multiple hobbies
* Multiple languages

These are **independent**.

This creates unnecessary combinations.

---

### Convert to 4NF

### Student Hobby

| Student | Hobby    |
| ------- | -------- |
| Rahul   | Cricket  |
| Rahul   | Football |

### Student Language

| Student | Language |
| ------- | -------- |
| Rahul   | Hindi    |
| Rahul   | English  |

Now no Multi-Valued Dependency exists.

---

## Cross Questions

### Q1. What does 4NF remove?

Multi-Valued Dependency.

### Q2. Which Normal Form is required before 4NF?

BCNF.

---

## ⭐ Interview Tip

4NF is **rarely used in interviews**, but you should know:

> **4NF removes Multi-Valued Dependencies.**

---

## 📌 Remember Keywords

* BCNF
* Multi-Valued Dependency
* Independent Attributes

---

## 📝 Revision

* 4NF = BCNF + No Multi-Valued Dependency.

5NF ensures this.

# Q54. What is Denormalization?

## Interview Answer

**Denormalization** is the process of **combining normalized tables to improve read performance**, even though it introduces some data redundancy.

---

## Detailed Explanation

Normalization splits data into many tables.

This reduces redundancy but increases the number of **JOIN** operations.

For applications where **read operations are much more frequent**, we intentionally combine tables.

This is called Denormalization.

---

### Example

Normalized:

Employee

| EmpID | Name | DeptID |
| ----- | ---- | ------ |

Department

| DeptID | DeptName |
| ------ | -------- |

To display employee details, we need a JOIN.

After denormalization:

| EmpID | Name  | DeptName |
| ----- | ----- | -------- |
| 1     | Rahul | Sales    |
| 2     | Aman  | HR       |

No JOIN is needed.

Reads become faster.

---

## Advantages

* Faster SELECT queries.
* Fewer JOIN operations.
* Better reporting performance.

---

## Disadvantages

* Duplicate data.
* More storage.
* Update anomalies may return.

---

## Real-Life Example

E-commerce websites often denormalize product data to display product listings quickly.

---

## Cross Questions

### Q1. Why is Denormalization used?

To improve read performance.

### Q2. Does it increase redundancy?

**Yes.**

---

## ⭐ Interview Tip

Remember:

* **OLTP systems** (banking, order processing) → Prefer **Normalization**.
* **OLAP/reporting systems** (analytics, dashboards) → Often use **Denormalization** for faster reads.

---

## 📌 Remember Keywords

* Performance
* JOIN
* Read Speed
* Redundancy

---

## 📝 Revision

* Combines tables.
* Faster reads.
* More redundancy.
* Used for performance optimization.

# Q56. What is Functional Dependency (FD)?

## Interview Answer

A **Functional Dependency (FD)** is a relationship between two attributes in a table where **one attribute uniquely determines another attribute**.

It is represented as:

> **X → Y**

which means **X determines Y**.

---

## Detailed Explanation

Suppose we have:

| RollNo | Name  | Branch |
| ------ | ----- | ------ |
| 101    | Rahul | CSE    |
| 102    | Aman  | IT     |

Here,

```
RollNo → Name
RollNo → Branch
```

Why?

Because if you know **RollNo**, you can uniquely determine the student's Name and Branch.

But,

```
Name → RollNo
```

is **not always true**, because two students can have the same name.

---

## Real-Life Example

Think of **Aadhaar Number**.

If I know your Aadhaar Number,

I can find:

* Name
* DOB
* Address

So,

```
Aadhaar → Name
Aadhaar → DOB
Aadhaar → Address
```

---

## Cross Questions

### Q1. What does X → Y mean?

X uniquely determines Y.

### Q2. Can one attribute determine multiple attributes?

**Yes.**

Example:

```
RollNo → Name, Branch, Age
```

### Q3. Does Y always determine X?

**No.**

---

## ⭐ Interview Tip

Remember:

A Functional Dependency is about **uniqueness**, not about the number of rows.

---

## 📌 Remember Keywords

* Determines
* Unique
* X → Y
* Dependency

---

## 📝 Revision

* X → Y means X determines Y.
* Functional Dependency is used in Normalization.
* Helps identify redundancy.

# Q57. Types of Functional Dependency

There are **four important types** asked in interviews:

1. Trivial FD
2. Non-Trivial FD
3. Fully Functional Dependency
4. Transitive Dependency

---

## 1. Trivial Functional Dependency

### Interview Answer

A Functional Dependency is **Trivial** if the right-hand side attribute is already a subset of the left-hand side.

```
(A, B) → A
```

Here,

A is already present on the left side.

Hence,

It is Trivial.

---

### Example

```
(StudentID, Name) → StudentID
```

---

### Interview Tip

If RHS is already inside LHS,

👉 **Trivial FD**

---

## 2. Non-Trivial Functional Dependency

### Interview Answer

A Functional Dependency is **Non-Trivial** if the right-hand side attribute is **not** present in the left-hand side.

Example:

```
RollNo → Name
```

RollNo does not contain Name.

Hence,

Non-Trivial FD.

---

## 3. Fully Functional Dependency

### Interview Answer

An attribute is **Fully Functionally Dependent** if it depends on the **entire Primary Key**, not on part of it.

---

### Example

Composite Key:

(StudentID, CourseID)

```
(StudentID, CourseID) → Marks
```

Marks depends on both StudentID and CourseID.

This is Fully Functional Dependency.

---

## 4. Transitive Dependency

### Interview Answer

A Transitive Dependency exists when

```
A → B
B → C
```

Therefore,

```
A → C
```

indirectly.

---

### Example

```
EmpID → DeptID
DeptID → DeptName
```

Therefore,

```
EmpID → DeptName
```

This is Transitive Dependency.

---

## ⭐ Interview Tip

This is exactly the dependency removed in **3NF**.

---

## 📌 Remember Keywords

* Trivial
* Non-Trivial
* Fully Functional
* Transitive

---

## 📝 Revision

* Trivial → RHS ⊆ LHS
* Non-Trivial → RHS not in LHS
* Fully Functional → Depends on whole key
* Transitive → Indirect dependency

# Q58. What are Armstrong's Axioms?

## Interview Answer

**Armstrong's Axioms** are a set of inference rules used to derive new Functional Dependencies from existing ones.

There are **three basic axioms**:

1. Reflexivity
2. Augmentation
3. Transitivity

---

## 1. Reflexivity

If

```
Y ⊆ X
```

then

```
X → Y
```

Example:

```
(StudentID, Name) → StudentID
```

---

## 2. Augmentation

If

```
X → Y
```

then

```
XZ → YZ
```

Example:

```
RollNo → Name
```

Then,

```
(RollNo, Branch) → (Name, Branch)
```

---

## 3. Transitivity

If

```
X → Y
Y → Z
```

then

```
X → Z
```

Example:

```
EmpID → DeptID
DeptID → DeptName
```

Then,

```
EmpID → DeptName
```

---

## Cross Questions

### Q1. How many basic Armstrong's Axioms are there?

**Three.**

### Q2. Which axiom is most commonly used?

**Transitivity.**

---

## ⭐ Interview Tip

Generally, placement interviews ask only the names and simple examples of the three basic axioms.

---

## 📌 Remember Keywords

* Reflexivity
* Augmentation
* Transitivity
* Inference Rules

---

## 📝 Revision

* Reflexivity → Subset.
* Augmentation → Add same attributes.
* Transitivity → Chain rule.

# Q59. What is Attribute Closure?

## Interview Answer

The **Closure of an attribute set** is the set of **all attributes that can be functionally determined** from that attribute set.

It is denoted as:

```
X⁺
```

---

## Why is Attribute Closure Used?

It helps us:

* Find Candidate Keys.
* Check Functional Dependencies.
* Verify Normal Forms.

---

## Example

Suppose:

```
A → B
B → C
```

Find **A⁺**.

Start:

```
A⁺ = {A}
```

Using

```
A → B
```

Add B.

```
A⁺ = {A, B}
```

Using

```
B → C
```

Add C.

Final:

```
A⁺ = {A, B, C}
```

---

## Cross Questions

### Q1. What does X⁺ mean?

Closure of X.

### Q2. Why do we calculate Closure?

To identify Candidate Keys and verify dependencies.

---

## ⭐ Interview Tip

If the closure contains **all attributes of the relation**, then that attribute set is a **Super Key**.

If it is also minimal, then it is a **Candidate Key**.

---

## 📌 Remember Keywords

* Closure
* Candidate Key
* Super Key
* Functional Dependency

---

## 📝 Revision

* Closure = All attributes determined by X.
* Used for keys and normalization.

# Q60. Lossless vs Lossy Decomposition

## Interview Answer

**Decomposition** means breaking a table into smaller tables.

There are two types:

* **Lossless Decomposition**
* **Lossy Decomposition**

---

## 1. Lossless Decomposition

### Definition

After joining the decomposed tables, **all original data is recovered exactly**.

No information is lost.

### Example

Student:

| StudentID | Name | DeptID |
| --------- | ---- | ------ |

↓

Student

| StudentID | Name |

Department

| DeptID | DeptName |

Join them back using DeptID.

You get the original table.

This is **Lossless**.

---

## 2. Lossy Decomposition

### Definition

After joining the tables, the original data **cannot be reconstructed correctly**, or extra/spurious tuples appear.

Information is lost or incorrect combinations are produced.

---

## Cross Questions

### Q1. Which decomposition is preferred?

**Lossless Decomposition.**

### Q2. Why?

Because no information is lost.

### Q3. Can Lossy Decomposition create incorrect records?

**Yes.**

---

## ⭐ Interview Tip

A decomposition should ideally satisfy:

* **Lossless Join**
* **Dependency Preservation**

These are the two main goals while decomposing tables during normalization.

---

## 📌 Remember Keywords

* Lossless
* Lossy
* Join
* Dependency Preservation
* Information Loss

---

## 📝 Revision

* Lossless → Original table recovered.
* Lossy → Information lost or incorrect tuples generated.
* Always prefer Lossless Decomposition.

---

Think of a puzzle 🧩:

* **Lossless** → Break the puzzle and reassemble it perfectly.
* **Lossy** → After reassembling, some pieces are missing or in the wrong place.

# Q61. What is an Index in DBMS?

## Interview Answer

An **Index** is a special data structure that **improves the speed of data retrieval** from a database table.

It works like the **index of a book**, allowing the database to find records quickly without scanning the entire table.

---

## Detailed Explanation

Suppose an **Employee** table has **10 million records**.

You execute:

```sql
SELECT * FROM Employee WHERE EmpID = 5000;
```

Without an index:

* The database checks each row one by one.
* This is called a **Full Table Scan**.

With an index:

* The database directly jumps to the required record.
* Search becomes much faster.

---

## How Does an Index Work?

Imagine a book.

Without an index:

You read every page to find "Normalization."

With an index:

You look at the index page and directly go to page 143.

A database index works in exactly the same way.

---

## Real-Life Example

Think of your phone contacts.

Without the search feature, you'd scroll through thousands of contacts.

With the search bar (index), you type "Rahul" and get the result instantly.

---

## Cross Questions

### Q1. Why do we use an Index?

To improve query performance.

### Q2. Does an Index store actual table data?

No. It stores key values and pointers to the actual rows.

### Q3. Which SQL operation benefits the most?

**SELECT** queries.

---

## ⭐ Interview Tip

A common misconception:

> **Does an Index make every operation faster?**

❌ No.

Indexes make **SELECT** operations faster but can make **INSERT, UPDATE, and DELETE** slightly slower because the index also needs to be updated.

---

## 📌 Remember Keywords

* Fast Search
* Data Structure
* Pointer
* Query Optimization
* Full Table Scan

---

## 📝 Revision

* Index improves search speed.
* Similar to a book index.
* Mostly benefits SELECT queries.
* Has a maintenance cost for writes.

Query:

# Q63. Types of Index

## Interview Answer

The most common types of indexes are:

1. Primary Index
2. Secondary Index
3. Clustered Index
4. Non-Clustered Index
5. Composite Index

---

## 1. Primary Index

Created on the **Primary Key**.

Example:

```sql
PRIMARY KEY(EmpID)
```

Since EmpID is unique, searching becomes very fast.

---

## 2. Secondary Index

Created on a **non-primary column**.

Example:

```sql
CREATE INDEX idx_email
ON Employee(Email);
```

Used when searching frequently by Email.

---

## 3. Composite Index

Created using multiple columns.

Example:

```sql
CREATE INDEX idx_name_city
ON Employee(Name, City);
```

Useful when queries frequently filter on both columns.

---

## Real-Life Example

If you often search employees by **Department + City**, a composite index on `(Department, City)` can speed up those queries.

---

## Cross Questions

### Q1. Can a table have multiple indexes?

**Yes.**

### Q2. Can we create an index on a non-primary column?

**Yes.**

### Q3. What is a Composite Index?

An index created on multiple columns.

---

## ⭐ Interview Tip

The **order of columns matters** in a Composite Index.

Example:

```sql
INDEX(Name, City)
```

helps:

```sql
WHERE Name='Rahul'
```

and

```sql
WHERE Name='Rahul' AND City='Delhi'
```

But it generally **does not help** much for:

```sql
WHERE City='Delhi'
```

This is called the **Leftmost Prefix Rule**.

---

## 📌 Remember Keywords

* Primary Index
* Secondary Index
* Composite Index
* Leftmost Prefix Rule

---

## 📝 Revision

* Primary → Primary Key.
* Secondary → Other columns.
* Composite → Multiple columns.
* Column order matters.

# Q64. Clustered vs Non-Clustered Index

## Clustered Index

### Interview Answer

A **Clustered Index** stores the **actual table data in the order of the index**.

Since the data itself is sorted, a table can have **only one Clustered Index**.

---

## Example

Suppose records are physically stored like:

```text
101
102
103
104
105
```

The data itself is ordered by the indexed column.

---

## Advantages

* Very fast range queries.
* Fast sorting.
* Fast sequential access.

---

## Non-Clustered Index

### Interview Answer

A **Non-Clustered Index** stores **only the index values and pointers to the actual rows**.

The actual table data remains in its original order.

A table can have **multiple Non-Clustered Indexes**.

---

## Example

```text
Index
101 → Row 500
102 → Row 3
103 → Row 120
```

The index points to where the data is stored.

---

## Real-Life Example

Think of a textbook:

* **Clustered Index** → Chapters are already arranged in order.
* **Non-Clustered Index** → The index page tells you which page to jump to.

---

## Cross Questions

### Q1. How many Clustered Indexes can a table have?

Only **one**.

### Q2. How many Non-Clustered Indexes can a table have?

Multiple.

### Q3. Which one stores actual data in sorted order?

Clustered Index.

---

## ⭐ Interview Tip

In **MySQL InnoDB**, the **Primary Key is implemented as a Clustered Index**. If no Primary Key exists, InnoDB chooses another suitable unique key or creates a hidden row ID.

---

## 📌 Remember Keywords

* Clustered
* Non-Clustered
* Sorted Data
* Pointer
* Physical Storage

---

## 📝 Revision

* Clustered → Data stored in index order.
* Non-Clustered → Index stores pointers.
* One Clustered Index.
* Many Non-Clustered Indexes.

---

# 🔥 Interview Comparison

| Clustered Index            | Non-Clustered Index            |
| -------------------------- | ------------------------------ |
| Data stored in index order | Separate index with pointers   |
| Only one per table         | Multiple allowed               |
| Faster for range queries   | Great for lookups              |
| Changes physical order     | Does not change physical order |

---

# 🎯 Memory Trick

Think of a library:

* 📚 **Clustered Index** → Books are physically arranged by roll number.
* 📖 **Non-Clustered Index** → Catalog tells you which shelf the book is on.

# Q65. What is a B+ Tree Index?

## Interview Answer

A **B+ Tree Index** is a balanced tree data structure used by most databases (like MySQL InnoDB) to store indexes efficiently.

It provides **fast searching, insertion, deletion, and range queries**.

---

## Detailed Explanation

Imagine Employee IDs:

```
1001
1002
1003
...
1000000
```

Instead of checking one by one, the B+ Tree organizes them into levels.

```
           5000
        /         \
    <5000       >=5000
```

The database traverses the tree from root to leaf.

Search complexity becomes approximately:

> **O(log n)**

instead of **O(n)**.

---

## Why B+ Tree?

* Fast search.
* Fast insert/delete.
* Efficient range queries.
* Leaf nodes are linked together for sequential traversal.

---

## Real-Life Example

Think of a dictionary.

You don't read every word.

You quickly jump to the required section.

That's how a B+ Tree works.

---

## Cross Questions

### Q1. Why is B+ Tree preferred over Binary Search Tree?

Because it remains balanced and is optimized for disk storage.

### Q2. Is B+ Tree good for range queries?

**Yes. It's one of its biggest advantages.**

---

## ⭐ Interview Tip

Most relational databases (especially **MySQL InnoDB**) use **B+ Trees** for indexes by default.

---

# Q66. What is a Hash Index?

## Interview Answer

A **Hash Index** uses a **hash function** to map a key directly to its storage location.

It is extremely fast for **exact match searches**, but not suitable for range queries.

---

## Detailed Explanation

Suppose:

```
EmpID = 105
```

Hash Function:

```
hash(105) → Bucket 7
```

The database directly checks Bucket 7.

No tree traversal is required.

---

## Advantages

* Very fast equality search.
* O(1) average lookup.

---

## Disadvantages

* Cannot efficiently perform:

```
WHERE salary > 50000
```

or

```
BETWEEN
ORDER BY
```

---

## Cross Questions

### Q1. Which is better for range queries?

**B+ Tree**

### Q2. Which is faster for exact match?

**Hash Index**

---

## ⭐ Interview Tip

Easy memory trick:

* **Hash → "="**
* **B+ Tree → ">, <, BETWEEN"**

# Q67. What is Query Optimization?

## Interview Answer

**Query Optimization** is the process of choosing the most efficient execution plan to execute a SQL query.

The database optimizer decides the fastest way to retrieve the required data.

---

## Detailed Explanation

Example:

```sql
SELECT *
FROM Employee
WHERE Email = 'abc@gmail.com';
```

The optimizer decides:

* Use Index?
* Full Table Scan?
* Which Join Order?
* Which Join Algorithm?

The plan with the lowest estimated cost is selected.

---

## How to Optimize Queries?

* Create proper indexes.
* Avoid `SELECT *`.
* Filter early using `WHERE`.
* Use appropriate JOINs.
* Avoid unnecessary subqueries.
* Analyze execution plans.

---

## Cross Questions

### Q1. Who optimizes a query?

The **Query Optimizer** inside the DBMS.

### Q2. Can indexes help optimization?

**Yes.**

---

## ⭐ Interview Tip

Good SQL writing + proper indexing = faster query execution.

Security and simplification.
